## Assumptions
- 「このプロジェクトに活かせる部分」は、運用手順・テンプレート・検証フローへの適用可能性を指す。
- 計画書は日本語で作成し、命名規則は JST 時刻入り形式に従う。
- 不明語や手法名は一次情報（公式ドキュメント等）を優先して確認する。

## Hypotheses
- H1: Harness engineering の主要概念は、既存の `PLAN -> TASKS -> REPORT` 運用の品質ゲート強化に転用できる。
- H2: 外部一次情報で裏付けた項目のみ計画へ採用することで、過剰提案や不適合提案を減らせる。

## Research Plan
- Round 1 Query:
  - 入力文書内の不明概念・手法を特定し、公式情報で定義と推奨実装を確認する。
- Round 2 Query:
  - 候補施策ごとの導入コスト/効果/リスクを比較し、優先順位を確定する。
- Exit Criteria:
  - 主要仮説ごとに支持/反証の根拠がある
  - 未解決論点に次アクションがある

## Approach
- 入力文書を論点分解し、適用可能性（高/中/低）で仕分ける。
- 不明点はラウンド制の Web 調査で一次情報を取得し、REPORT の Evidence Record に記録する。
- 採用施策を短期/中期に分け、担当アクション・成功指標・ロールバック方針を含む計画書に落とし込む。
- 標準フロー: `PLAN -> Web検索(不足知識) -> TASKS -> 実行 -> REPORT`

## Definition of Done
- 入力文書の要点とプロジェクト適用可否が整理されている。
- 不明点に対する Web 調査証跡が REPORT に記録されている。
- `docs/plans/{yyyy-mm-dd}_{HHMMSS}_*.md` に改善計画書が作成されている。
- `docs/PROJECT_CONTEXT.md` に新規知見が反映され、`docs/history/` に履歴が追記されている。
- 品質ゲート（少なくとも `scripts/verify`）結果が REPORT に記録されている。

## Risks / Unknowns
- 入力文書の主張がこのリポジトリの運用前提と一致しない可能性。
- Web 上の二次情報偏重による誤適用リスク（公式情報優先で低減）。
- 文書中心変更のため、実装効果は次フェーズで追加検証が必要。
